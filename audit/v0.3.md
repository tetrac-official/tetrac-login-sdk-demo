# Migration PRD — `@tetrac/login-sdk` v0.2.0 → v0.3.0

**Status:** Draft · target SDK release `0.3.0`
**Date:** 2026-06-12
**Audience:** Developers integrating `@tetrac/login-sdk` (this demo is the reference integration).
**Scope:** What breaks between the published `0.2.0` and the hardened `0.3.0`, how to migrate an
integration, and — because **an SDK can only secure the layers it owns** — the production setup the
_developer_ must provide around it.

---

## 0. TL;DR

`0.3.0` is a **security-hardening release with hard breaks and no data migration** (by design — the SDK
is pre-production). Two things change that you cannot paper over:

1. **Auth model:** email/biometric accounts no longer send a `passkeyHash`. The server now stores an
   **ed25519 auth public key** and login is **challenge → sign → verify**. The server holds _only public
   keys_ — no passkey-derived secret of any kind.
2. **Encryption:** wallet secrets move from crypto-es **AES‑CBC** (no auth tag) to Web Crypto
   **AES‑256‑GCM** (authenticated). The ciphertext string format changes.

Because both the **stored record shape** and the **ciphertext format** change, **existing accounts and
encrypted wallets from `0.2.0` are unreadable under `0.3.0`.** There is **no dual-format / migration
path** — you **wipe the store and re-register**. For a demo/dev deployment that is a one-line reset.

The good news: the **client method signatures are unchanged** (`registerWithEmail({ email, passkey })`,
`loginWithEmail(...)`, `useAuth()`, `<AuthProvider>` …). Most integration _source_ does not move — the
break is in _data_ and in _documentation that describes the old flow_.

---

## 1. The shared-responsibility model (read this first)

The SDK runs **inside your app** (your Next.js server, your frontend bundle). It cannot see your network,
your TLS terminator, or the TCP source of a request. So security splits in two. Trying to make the SDK do
the developer's half is the single most common way to get a false sense of safety.

| Concern                                                      | Owned by **the SDK** (`0.3.0`)                     | Owned by **you, the developer / infra**                                |
| ------------------------------------------------------------ | -------------------------------------------------- | ---------------------------------------------------------------------- |
| Client-side key derivation (PBKDF2 / PRF / SHA-256(sig))     | ✅                                                 | —                                                                      |
| Wallet secret encryption (AES‑256‑GCM, authenticated)        | ✅                                                 | —                                                                      |
| Challenge issuance, single-use consume, signature verify     | ✅                                                 | —                                                                      |
| Session token issue / validate / single-active-session       | ✅                                                 | —                                                                      |
| Application rate-limit **counters** (per-target buckets)     | ✅                                                 | —                                                                      |
| Input validation, key-namespace isolation, fail-safe parsing | ✅                                                 | —                                                                      |
| **Real client IP identity**                                  | ⚠️ _only if you set `trustProxyHeaders` correctly_ | ✅ **You** terminate the proxy and tell the SDK how many hops to trust |
| **Volumetric / network DoS** (L3/L4, request floods)         | ❌ can't                                           | ✅ Edge WAF / firewall / autoscale                                     |
| **TLS, HSTS, secure transport**                              | ❌                                                 | ✅ Your platform / reverse proxy                                       |
| **XSS containment** (token + appKey live in the tab)         | partial (memory-only default)                      | ✅ CSP, Trusted Types, SRI, dep hygiene                                |
| **Bot / abuse on open endpoints** (register, challenge)      | ❌                                                 | ✅ CAPTCHA / Turnstile, allow-lists                                    |
| **Email ownership / account recovery**                       | ❌ out of scope                                    | ✅ Your email-verification + recovery flow                             |
| **Storage durability + atomic `getDel`**                     | depends on adapter                                 | ✅ Pick a backend that guarantees it (see §6)                          |
| **Secrets management** (backend creds)                       | ❌                                                 | ✅ Your env / secret store                                             |

> **The one-line version:** the SDK makes each _request_ safe to process; **you** make the _perimeter_
> safe and supply the _identity of the caller_. The most important thing the SDK genuinely **cannot** do
> alone is **per-source rate limiting**, because without a trusted proxy it has no reliable client IP.

## 2. Breaking changes (the `0.2.0 → 0.3.0` delta)

Each item: **what changed · why · impact · what you do.** Items marked **[landed]** are already in the
working tree; **[planned]** are scheduled for `0.3.0` and listed so the migration is complete on release.

### 2.1 `passkeyHash` removed → ed25519 signature auth **[landed]**

- **What:** Email & biometric accounts derive an ed25519 _auth keypair_ from the app key
  (`seed = SHA‑256("ttc-auth-v1:" + appKey)`). Registration sends `authPublicKey`; login is
  `POST /challenge → sign(challenge) → POST /login { signature }`. `UserData.passkeyHash` is gone;
  `UserData.authPublicKey` replaces it.
- **Why:** A stored `passkeyHash` was a server-held secret-equivalent and a bearer credential
  (`hash(passkey) === passkeyHash` was a login). The server now stores **only public keys** and verifies a
  fresh signature — no replayable secret on the server, ever.
- **Impact:** **Hard break.** Old records (`passkeyHash`, no `authPublicKey`) cannot log in. The login wire
  shape changes from `{ email }` to `{ email, signature, challenge }`.
- **You do:** Re-register accounts (§5). No source change if you use `loginWithEmail`/`useAuth` — the SDK
  does the challenge/sign internally.

### 2.2 AES‑CBC (crypto-es) → AES‑256‑GCM (Web Crypto) **[landed]**

- **What:** `encryptSecret` returns `b64url(iv) + ":" + b64url(ciphertext‖tag)` via `crypto.subtle`.
  Decryption is now **async** and **fails closed** on a bad tag.
- **Why:** CBC had **no authentication tag** — ciphertext was malleable/tamperable. GCM is authenticated.
- **Impact:** **Hard break.** `0.2.0` ciphertext does not decrypt under `0.3.0`. `decryptSecret` is now a
  Promise (it already was awaited in the SDK; no integrator change unless you called crypto directly).
- **You do:** Nothing in source if you only use wallet helpers; the re-register in §5 produces new GCM
  ciphertext.

### 2.3 `securityLevel` replaces hardcoded iterations; per-user pinning **[landed]**

- **What:** New `config.securityLevel: 1 | 2 | 3` → PBKDF2-HMAC-SHA256 `{1:100k, 2:600k, 3:1M}`. The
  **resolved count** is pinned per account in `UserData.pbkdf2Iterations` and returned with the challenge,
  so derivation stays stable for an account even if you change the default later. **Default = `2` (600k,
  OWASP-2023 minimum).**
- **Why:** `0.2.0` used a single hardcoded iteration count; this lets the integrator trade login latency
  vs brute-force resistance, and makes the choice durable per user.
- **Impact:** Non-breaking config addition. Affects **email/passkey** accounts only.
- **You do:** Optionally set `securityLevel`. Level `2` is the recommended default. Level `1` is below
  OWASP and for legacy/low-value only.

### 2.4 Rate-limiting model: per-target buckets + penalize-on-failure + proxy hops **[planned]**

- **What:** (a) When `trustProxyHeaders` is false the SDK **stops gating on the shared `"unknown"` IP
  bucket** and rate-limits `/challenge` and `/search-wallet` **per `publicKey`**. (b) `login` / `login-wallet`
  **verify first, then increment the counter only on _failure_** — a _valid_ login is never throttled.
  (c) New `config.trustedProxyHops` selects the **rightmost** `x-forwarded-for` hop after N trusted proxies
  (spoof-resistant) when `trustProxyHeaders` is true.
- **Why:** In `0.2.0` every client collapsed into one global `"unknown"` bucket (one abuser locked out
  _everyone_), and the failure counter incremented _before_ auth (an attacker who knew your email could
  lock out your _correct_ login).
- **Impact:** Behavior change, not a source break. **Residual you must own:** without `trustProxyHeaders`,
  limiting is **per-target, not per-source** — an abuser can still DoS _one_ target's challenge issuance.
  Per-_source_ limiting requires a trusted proxy (see §3 / §6).
- **You do:** Set `trustProxyHeaders` + `trustedProxyHops` for your deployment (§3), and put real
  per-source limiting at the edge.

### 2.5 Input validation + record-bloat cap + key-namespace isolation **[landed]**

- **What:** `register` / `challenge` / `login-wallet` reject malformed `publicKey` (whitespace, >128 chars)
  and `email` (regex, ≤320) with `400`. `import-wallet` is capped at `config.maxWalletsPerUser` (default
  **64**). Session tokens live under a disjoint `session:` prefix (was `pubKey:session:`), and
  `getUserByPublicKey` guards `JSON.parse` (malformed value → `null`, not a crash).
- **Why:** Closes unvalidated-input, unbounded-import record bloat, an attacker-chosen-`publicKey`
  namespace collision, and a `JSON.parse` crash (DoS).
- **Impact:** Stricter `400`s on previously-accepted junk. `maxWalletsPerUser` may reject very large
  imports.
- **You do:** Ensure you don't rely on accepting malformed keys; tune `maxWalletsPerUser` if 64 is low.

### 2.6 Memory-only vault (the `appKeyStorage` config is removed) **[landed]**

- **What:** The app key is held in **memory only** — never `sessionStorage`, never `localStorage`, never
  the network. The `appKeyStorage` config option (and its `"session"` mode) is **removed**: a reload (or
  crash) always drops the key and forces re-auth. `autoLockMs` (idle auto-lock) and `lockOnHide` are
  honored, and the vault now also locks on `pagehide` / page `freeze`, and re-locks on a bfcache restore
  (`pageshow` `persisted`) — closing the reload-revives-the-key gap (CLIENTVAULT-3/4/5).
- **Why:** A raw key in `sessionStorage` was XSS-readable at rest _and_ let a reload reset the idle window
  to keep the key hot indefinitely. Removing the mode deletes the footgun outright.
- **Impact:** **Breaking if you set `appKeyStorage`.** Remove it — `appKeyStorage: "session"` no longer
  exists and `"memory"` is now the only (implicit) behavior. A page reload always re-authenticates.
- **You do:** Delete any `appKeyStorage` from your config. If you want reload-survival _without_ re-typing
  a passkey, use **biometric unlock** (Touch/Face-ID-gated wrapped blob) — the secure path — instead of
  persisting a raw key.

### 2.7 New biometric **unlock** (PRF-wrapped local vault) **[landed]**

- **What:** `enableBiometricUnlock` / `unlockViaBiometric` wrap the app key with an HKDF/AES-GCM key
  derived from a WebAuthn PRF secret, stored in an IndexedDB `unlock_blobs` store, gated by Touch/Face ID.
- **Why:** Lets a returning user re-arm the vault with biometrics instead of re-deriving from passkey.
- **Impact:** Additive. New `useBiometricUnlock()` hook.
- **You do:** Optional — wire the hook if you want one-touch unlock.

### 2.7b Session, cross-tab & storage hardening **[landed]**

- **What:**
  - **Cross-tab lock (CLIENTVAULT-7):** logout/lock in one tab now locks sibling tabs of the same origin
    (via a `storage` event — logout removes the shared token; an explicit lock bumps a sentinel key).
  - **`useExportKey` auto-clear default ON (CLIENTVAULT-8):** revealed plaintext clears after **30s** by
    default (pass `autoClearMs: 0` to disable).
  - **Logout uses `keepalive: true` (AUTHSESSION-10):** the `POST /logout` revocation lands even when
    logout coincides with the page unloading.
  - **Default `sessionTtlSeconds` 24h → 4h (AUTHSESSION-10):** a leaked bearer token dies sooner; every
    login still revokes the prior token (single active session), so "revoke all" is inherent to logout.
  - **Production storage guard (SERVERSIDE-10 / WI-10):** `resolveStorageAdapter()` now **throws** when
    `NODE_ENV==='production'` and no backend env is configured, instead of silently connecting to
    `redis://localhost:6379` (which would give each instance its own ephemeral store). Partial Upstash
    config (URL without TOKEN) logs a warning.
- **Impact:** Mostly non-breaking behavior changes. **Two to note:** (1) sessions now expire after **4h**
  by default — set `sessionTtlSeconds` higher if you relied on 24h; (2) a **production deploy with no
  storage backend now fails fast at startup** — wire up `REDIS_URL` / Vercel KV / Upstash (this is a
  feature: it surfaces a misconfiguration that used to fail silently).
- **You do:** Configure a real storage backend in production (you already should — see §6.3); optionally
  tune `sessionTtlSeconds`.

### 2.7c Domain-separated app-key derivation (`appId`) **[landed]**

- **What:** A new required-with-default **`appId`** config domain-separates app-key derivation on both
  paths: the email/passkey PBKDF2 salt becomes `SHA-256(appId : email)` (was the bare email), and the
  message a Web3 wallet signs to derive its key is now **appId-bound** (was a fixed constant). So the same
  `(email+passkey)` — or the same wallet — derives a **different app key per app** (CRYPTO-2 / H4).
- **Why:** With a bare-email salt and a constant wallet message, the same user derived the _same_ app key
  on **any** deployment built on this SDK → cross-app key reuse (a key cracked or coerced on one app
  unlocks the same user on another), and one precomputed table worked across all deployments.
- **Impact:** **Breaking (clean break, no migration):** changing the derivation changes every app key, so
  `0.2.0` ciphertext won't decrypt — consistent with the "wipe + re-register" stance (§5.2). `appId`
  **defaults to `"ttc"`**, which works but gives **no** cross-app isolation; the client logs a one-time
  warning until you override it.
- **You do:** Set a **unique, stable** `appId` per deployment (e.g. your product/domain) in your client
  config. It must never change for a live deployment — changing it re-derives all keys and orphans
  existing encrypted wallets.

### 2.7d Dependency: `crypto-es` removed → `@noble/hashes` **[landed]**

- **What:** Key derivation + hashing (PBKDF2-HMAC-SHA256, SHA-256) moved off `crypto-es` to
  **`@noble/hashes`** (a sync, audited, minimal lib already in the tree via `@solana/web3.js`/`viem`).
  AES was already on Web Crypto. `crypto-es` is no longer a dependency.
- **Impact:** **None for integrators** — internal, **byte-identical** output (verified), so no key change
  and no API change. Bonus: meaningfully faster PBKDF2.
- **You do:** Nothing. (Your package manager will install `@noble/hashes` transitively as before.)

### 2.7e Build/supply-chain hygiene + docs (WI-11 / WI-13) **[landed]**

- **What:** single lockfile (`yarn.lock` dropped), `tsup sourcemap:false` (no source maps in the
  published tarball), verified `peerDependenciesMeta` (storage adapters / `next` / `react` optional),
  CI (`.github/workflows/ci.yml`: typecheck + test + build + `npm audit`) + `dependabot.yml`, a
  copy-paste **CSP** in the demo (`next.config.mjs headers()`) + a "Hardening the host app" README
  section, and a full README refresh (accurate endpoints, config incl. `appId`/`securityLevel`,
  Solana-vs-EVM chain scope, AES-256-GCM/`@noble` security model — removing the stale `crypto-es`/
  `passkeyHash`/`appKeyStorage` claims).
- **Impact:** None for integrators (build/docs only). `npm audit` = 3 _moderate_ in
  `@solana/web3.js → jayson` (transitive peer, below the high gate).
- **You do:** Nothing — but for a real publish you still need `npm publish --provenance` from CI
  (needs your CI + npm token) and Dependabot enabled on the GitHub repo (committing the file does that).

### 2.8 Planned for `0.3.0` (track before GA) **[planned]**

- Real **server-side WebAuthn assertion verification** (`@simplewebauthn/server`) so biometric login
  proves a hardware assertion, not just a derived signature (WI-15 — deliberately deferred).
- Server-side passkey **recovery** flow (WI-16).
- Publish via CI with `--provenance` (the only WI-11 item that can't be done in-repo).

> These do not change the migration mechanics below, but a production launch should wait for the WebAuthn
> assertion item (2.8) if biometric is a primary factor.

## 3. Telling the SDK who the caller is (`trustProxyHeaders` / `trustedProxyHops`)

This is the developer's half of rate-limiting. The SDK cannot read the TCP peer address — only your edge
knows it. Configure per environment:

```ts
// app/api/auth/[...action]/route.ts
export const { GET, POST } = createNextAuthRoutes({
  storage,
  config: {
    // Trust the proxy ONLY where a proxy you control sets the header.
    trustProxyHeaders: !!process.env.VERCEL, // demo already does this
    // How many proxies sit in front of you. The SDK takes the rightmost XFF hop
    // AFTER skipping this many trusted hops, so a client-spoofed leftmost entry
    // is ignored. Vercel/most single-edge setups: 0 (rightmost = real client).
    trustedProxyHops: 0,
  },
});
```

| Environment          | `trustProxyHeaders` | `trustedProxyHops`       | Notes                                                                      |
| -------------------- | ------------------- | ------------------------ | -------------------------------------------------------------------------- |
| Local dev (no proxy) | `false`             | —                        | No real IP; per-target limiting only. Fine for dev.                        |
| Vercel               | `true`              | `0`                      | Vercel's edge sets a trustworthy XFF; rightmost = visitor.                 |
| Cloudflare → origin  | `true`              | `0`–`1`                  | Prefer `CF-Connecting-IP` at the edge; set hops to match your chain.       |
| nginx/ALB → app      | `true`              | number of _your_ proxies | Only count proxies **you** control; never trust hops a client can prepend. |

**Setting `trustProxyHeaders: true` without an actual trusted proxy is a vulnerability** — a client can
forge `x-forwarded-for` and evade or misattribute limits. When in doubt, leave it `false` and rate-limit
at the edge.

## 5. Migration steps (integrator checklist)

### 5.1 Bump the dependency

```bash
npm install @tetrac/login-sdk@0.3.0
rm -rf .next            # clear stale HMR/build cache
npx tsc --noEmit        # catch any direct passkeyHash / sync-crypto usage
```

### 5.2 Wipe the store (no data migration by design)

`0.2.0` records carry a `passkeyHash` and crypto-es ciphertext that `0.3.0` cannot authenticate or decrypt.
There is intentionally **no dual-format migration**.

- **Memory adapter:** restart the process.
- **Redis / KV / Upstash:** flush the auth keyspace (`pubKey:*`, `email:*`, `session:*`, `challenge:*`,
  `ratelimit:*`) or the database if it's dedicated to auth.

All users **re-register** under the new signature-auth + GCM scheme. (Acceptable because the SDK is
pre-production; do **not** ship `0.3.0` to a base of real `0.2.0` accounts without an explicit re-onboard.)

### 5.3 Set proxy trust for each environment

Apply §3 to your route config.

### 5.4 Update the docs that describe the old flow _(this repo)_

The demo `README.md` still documents the removed model — fix it before publishing:

- The register sequence sends `{ email, passkeyHash, publicKeys, ciphertext }` → now
  `{ email, authPublicKey, publicKey, wallets[], pbkdf2Iterations }`.
- The login sequence shows `GET /user-data` + `hash(passkey) === passkeyHash` → now
  `POST /challenge → sign → POST /login { signature, challenge }`.
- "AES, crypto-es" → "AES‑256‑GCM (Web Crypto, authenticated)".
- "The server only holds: the **hash** of the passkey…" → "the server holds **only public keys** (an
  ed25519 auth public key + wallet public keys) and ciphertext — no passkey-derived secret."
- The `0x`/EVM note: external wallet **login** is Solana-only by design; EVM keypairs are internal signing
  wallets.

### 5.5 Re-run the demo

```bash
npm run dev
# Register fresh on each of the three flows; confirm login + a signed tx works.
```

## 6. Recommended production setup (the developer's half)

Layered, in priority order. Items 1–3 are non-negotiable for a public deployment.

1. **Terminate behind a trusted edge and configure proxy trust.** Vercel, Cloudflare, or nginx/ALB. Set
   `trustProxyHeaders: true` + correct `trustedProxyHops` **only** there (§3). This is what upgrades the
   SDK's limiting from per-target to per-source.
2. **Add edge/network rate limiting + WAF.** The SDK's counters are application-level and per-target;
   volumetric floods and per-source throttling belong at the edge (Vercel Firewall / Cloudflare Rate
   Limiting / nginx `limit_req`). Put a hard per-IP cap on `/api/auth/*`, a tighter one on `challenge`,
   `register`, `login`, `login-wallet`.
3. **Pick a storage backend with atomic `getDel` and durability.** Single-use challenges rely on an atomic
   read-and-delete. `MemoryAdapter` is **single-process only** (no good across serverless instances).
   Use Redis / Vercel KV / Upstash in production; ensure persistence if you don't want sessions to evaporate
   on restart.
4. **Contain XSS — the app key and session token live in the tab.** The SDK defaults the app key to
   memory-only, but a successful XSS during an unlocked window can still read it. Ship a strict **CSP**
   (no inline scripts, `script-src 'self'`), **Trusted Types**, **Subresource Integrity** on third-party
   scripts, and keep `autoLockMs` short. Audit your dependency tree.
5. **Gate the open endpoints against bots.** Email `register` has _no proof of email ownership_ and
   `challenge` is unauthenticated by nature. Add CAPTCHA / Cloudflare Turnstile (or an allow-list / invite
   gate) on `register` and, if abused, `challenge`.
6. **Own email verification + account recovery.** The SDK does not send email or verify ownership. If email
   is an identity, verify it (magic link / OTP) **before** registration, and design recovery
   (the planned server-side passkey recovery, 2.8, or your own).
7. **TLS + HSTS + secure headers everywhere.** Non-negotiable; the SDK assumes a secure transport.
8. **Keep secrets in a secret store**, not in client-reachable env. Only storage-backend creds belong
   server-side; nothing sensitive should reach the client bundle (the demo's `next.config.mjs` already
   externalizes backends).

## 8. Known residuals the SDK cannot close alone

An adversarial review of the rate-limit hardening confirmed these residuals. They are **inherent to an
SDK with no trusted source IP** and are the developer's edge/design responsibility (per §1 and §6) — not
SDK bugs. They are listed so you can decide which your deployment must mitigate.

| Residual                                                                                                                                                                            | Why it can't be fixed in the SDK alone                                                                                                                                                                                                                                                                                                             | Developer mitigation                                                                                                    |
| ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| **Per-target `/challenge` DoS** — flooding one victim's email/publicKey exhausts that target's challenge bucket.                                                                    | `/challenge` is unauthenticated; without a source IP the SDK can't tell the victim's own requests from the flood. Per-target limiting (which fixes the _global_ H5 lockout) is the trade.                                                                                                                                                          | Set `trustProxyHeaders` + edge per-IP limiting (§3, §6.2); optional CAPTCHA on `/challenge`.                            |
| **Account enumeration** via `/challenge` (known email → `200`+challenge, unknown → `400`).                                                                                          | A real login needs a challenge for known accounts and can't issue one for unknown — the distinction is structural. (`login` itself is non-revealing: all failures return a generic `401`.) Probes are now rate-limited (per-target, and per-IP when trusted) but distinct-email enumeration without a source IP is still bounded only at the edge. | Verify email ownership before registration (§6.6); edge limiting; optionally issue decoy challenges for unknown emails. |
| **Login work-amplification under flood** — `login` verifies (1 lookup + ed25519) before the failure counter increments, so a flood does cheap work per request even past the limit. | This is the cost of _verify-first_ (the AUTHSESSION-3 fix): a pre-auth counter keyed on email would re-introduce the victim lockout it was designed to prevent. The work is sub-millisecond.                                                                                                                                                       | Edge/volumetric limiting (§6.2); `trustProxyHeaders` so the per-source IP bucket gates the flood.                       |
| **Email squatting** — `register` has no proof of email ownership, so an attacker can claim an email first.                                                                          | The SDK sends no email and can't verify ownership.                                                                                                                                                                                                                                                                                                 | Verify email (magic link / OTP) **before** registration (§6.6); bot-gate `register` (§6.5).                             |
